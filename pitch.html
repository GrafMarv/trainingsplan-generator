Datenschutz-Architektur
Bereits umgesetzt
  • Spielerdaten ausschließlich in Supabase EU (Paris) — kein Drittlandtransfer für Speicherung
  • PDF-Extraktion client-seitig im Browser — Buchinhalte landen nie auf einem Server
  • Wissen nur als destillierte Chunks, nicht als Volltext gespeichert
  • Keine Spielernamen in Trainingsplan-Generierung (KI arbeitet nur mit Methodik)
  • Verein + Mannschaft als Organisationsstruktur, kein Vereins-Login nötig
Nächste Schritte
  • Pseudonymisierung für KI-Diagnostik-Analyse (Spieler-IDs statt Namen zu Claude)
  • Geburtsdatum → Jahrgang + Quartal (Q1–Q4) statt Vollgeburtsdatum
  • AVV mit Anthropic und OpenAI abschließen
  • Datenschutzerklärung + Einwilligungsformular für Eltern/Spieler
  • Optional: lokales Hosting (Synology NAS) für vollständige Datensouveränität
Pseudonymisierung — Wie es funktioniert
01
Echtdaten
Name, Jahrgang, Quartal in Supabase EU — verlassen nie den Server
02
Mapping
Server generiert Pseudonym-Code aus Spieler-ID (z.B. P-7F3A)
03
KI-Call
Nur Pseudonym + Messwerte gehen an Claude/OpenAI — keine Identität
04
Rückmapping
Server tauscht Pseudonyme zurück, Browser sieht echte Namen
// Was Claude sieht:
{ "id": "P-7F3A", "s10": 1.79, "s30": 4.02, "vmax_est": 24.1 }

// Was in Supabase EU bleibt:
{ "id": "P-7F3A" → "Lukas Meier", "jahrgang": 2009, "quartal": "Q2" }
